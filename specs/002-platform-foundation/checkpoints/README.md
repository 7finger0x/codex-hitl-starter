**Feature**: `002-platform-foundation`  
**Authority**: Constitution Article II, AGENTS.md, and the plan approved for task validation  
**Rule**: A checkpoint is pending unless its own record contains an explicit user decision for the exact reviewed scope. Approval of another artifact or checkpoint never carries over.

## Decision states

- `pending`: No decision has been supplied. Block all dependent work.
- `approved`: The exact recorded scope may proceed until its stated expiry or invalidation event.
- `approved_with_conditions`: Proceed only within the recorded conditions and compensating controls.
- `denied`: Do not proceed. Revise the proposal before requesting a new decision.
- `expired`: Prior approval is no longer valid.
- `superseded`: A later attributable decision replaces this record.

## Required decision record

Each checkpoint file must identify:

1. checkpoint ID and protected action;
2. exact files, versions, commands, target, remote, or artifacts in scope;
3. requirements and tasks served;
4. expected effects, irreversible impact, and data/content classification;
5. validation evidence and known gaps;
6. rollback or forward-recovery procedure;
7. requested decision and any options;
8. user-supplied decision, conditions, rationale, and timestamp;
9. invalidation conditions and expiry, if applicable; and
10. links to resulting verification evidence.

The agent must not populate approval identity, rationale, or status from inference.
Only explicit user input can change `pending` to an approval state.
